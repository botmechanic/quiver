# Arc OSS Answer Draft

Quiver’s OSS extraction is the per-second x402 streaming rail: a small, reusable loop that turns elapsed usage into discrete EIP-3009 authorizations, verifies each tick through Circle Gateway, and reconciles billing from `stream_events` instead of cached client counters. The reference `arc-nanopayments` starter and the `arc-commerce` / `arc-p2p-payments` repos all settle discrete, one-shot payments; none expose a streaming primitive. That gap — continuous, per-second settlement composed from batched authorizations — is what this extraction fills.

The first community adapter is Owncast. In local verification against Owncast `0.2.5`, the clean blog-post model of `USER_JOINED` / `USER_PARTED` was not exactly what fired: `USER_JOINED` was not observed, but `CHAT` provided the start signal with `eventData.user.id` and `eventData.clientId`; `USER_PARTED` fired and closed the same user; and `GET /api/integrations/clients` worked as a heartbeat fallback. Quiver’s sidecar uses that verified model to start the existing per-second x402 stream on Owncast chat presence, stop on `USER_PARTED` or heartbeat disappearance, and preserve `stream_events` as the source of truth.

Proof from the local adapter runs: clean-close session `25418b23-4f15-4146-aad8-3038320a011e` produced a verified `0.0001` USDC stream tick with Owncast user/client metadata in the raw ledger row. Heartbeat-fallback session `bc3fa644-031f-4bfa-89d1-7c4849aad1fb` was run with `USER_PARTED` withheld from the sidecar webhook; it stopped via `Owncast heartbeat missing` and wrote one verified `0.0001` USDC tick to `stream_events`. Archer and Scout remain the agent demo; Owncast is the creator-stack adapter proving the same rail can attach to a real OSS community surface.
